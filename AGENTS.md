# AGENTS.md

## 1. Codex 工作方式（Working Mode）

本项目中，Codex 应以“流程编排 + 文档生产助手”模式工作，遵循以下原则：

1. **阶段化执行**：按 01→07 的阶段推进，不跳步。
2. **模板优先**：优先复用 `templates/` 与 `.agents/skills/`，避免从零自由发挥。
3. **证据驱动**：每条结论应尽量回溯到输入材料来源。
4. **可审阅输出**：输出应结构化、可读、可评审、可归档。
5. **最小改动原则**：仅在必要文件内编辑，避免无关改动。

## 2. 项目目标（Project Goals）

根据输入资料，半自动形成以下七类输出：

1. 需求解析文档
2. 展厅叙事结构文档
3. 平面视觉策略文档
4. 展厅平面视觉系统文档
5. PPT 汇报 storyboard
6. 可生成 PPTX 的脚本基础
7. 归档说明与交付清单

## 3. 文件结构约定（Structure Rules）

- `input/`：仅放原始输入材料，不做最终输出。
- `output/`：仅放阶段产出和交付包。
- `templates/`：放标准模板，不混入真实项目内容。
- `.agents/skills/`：放技能说明，定义每阶段执行规范。
- `scripts/`：放自动化脚本；当前 `generate-ppt.js` 为占位实现。

## 4. 输出规范（Output Spec）

1. 文档主体语言为中文，必要术语可保留英文（如 storyboard、layout、grid）。
2. 每份文档应包含：背景/目标、输入依据、方法过程、结论输出、待确认项。
3. 内容应具备“可执行性”，避免空泛描述。
4. 文件命名建议包含阶段编号与主题。
5. 对外汇报内容应与内部策略文档保持口径一致。

## 5. 禁止事项（Do Not）

1. 不得杜撰输入材料中不存在的数据或事实。
2. 不得在未确认的情况下生成真实项目结论并标注为“最终版”。
3. 不得覆盖或污染 `input/` 原始资料。
4. 不得将敏感信息写入公开交付文档。
5. 不得跳过评审清单直接归档。

## 6. 验收标准（Acceptance Criteria）

以下条件全部满足方可视为合格：

1. 目录结构完整，且 01~07 阶段路径齐全。
2. 模板文件可直接复制使用，章节完整。
3. 每个 skill 提供明确流程与质量检查规则。
4. 脚本目录包含可扩展的 PPT 占位脚本与说明。
5. 文档语言、术语和命名风格统一。
6. 项目可被新成员在 30 分钟内理解并开始使用。
